# API-Rpi16GB

Backend server for the DHBW Anwendungsprojekt. Runs on a Raspberry Pi 5 (16 GB).

## Overview

This repository contains the backend infrastructure for an IoT sensor system.

Current data flow:

```
Sensor / Pico
    
    → MQTT over TLS
    → Mosquitto
    → Python Controller
    → HTTPS via nginx
    → Zig Backend API
    → PostgreSQL
```
The Controller no longer writes directly to PostgreSQL. Instead, it forwards validated MQTT sensor data to the Zig API through nginx. This keeps database access centralized in the backend.

## Stack

- **Zig 0.16.0** — HTTP API server
- **PostgreSQL 16** — sensor data + actuator command storage
- **nginx** — TLS termination, reverse proxy, rate limiting
- **Mosquitto** — MQTT broker, TLS, per-sensor ACL
- **Python controller** — MQTT → HTTPS API bridge
- **Python archiver** — daily job, moves rows older than 7 days to `sensor_data_archive`, purges archive after 3 years
- **Docker** — container orchestration, two isolated bridge networks (`app-net`, `sensor-net`)

## Architecture
### Previous flow

```
Sensor → MQTT Broker → Controller → PostgreSQL
```
### Current flow

```
Sensor → MQTT Broker → Controller → nginx → Zig Backend API → PostgreSQL
```

Why this was changed:
The old Controller had direct database access. That meant it needed database credentials and had to know database details.

The new design is cleaner:

- the Controller only handles MQTT input and API forwarding
- the Zig backend owns database access
- validation and persistence are centralized
- credentials are less widely distributed
- nginx provides a stable HTTPS entry point
- the system is easier to maintain and closer to a production-style architecture

## Controller Refactor

The Controller was changed from a direct database writer into an API client.

Removed from the Controller:

```
DB_HOST
DB_NAME
db_write_password
psycopg2 / direct PostgreSQL dependency
```
Added for the Controller:

```
API_BASE_URL=https://nginx
API_USERNAME=admin
api_password
ca_cert
```
The Controller now:
1. connects to Mosquitto via MQTT/TLS
2. receives messages from sensor topics such as sensor01/data
3. validates the payload
4. logs in to the Zig API through nginx
5. sends sensor readings to /api/v1/sensor-data
6. lets the backend store the data in PostgreSQL

This keeps the Controller focused on MQTT handling and moves persistence into the backend.

## Docker Networks

Two Docker bridge networks are used.

### sensor-net

Used for MQTT communication.

```
Mosquitto ↔ Controller
```
### app-net

Used for backend and API communication.

```
Controller ↔ nginx ↔ Backend ↔ PostgreSQL
```

This limits unnecessary cross-service communication and improves network isolation.

```
db_password.txt

jwt_secret.txt
```

## Deploy (on the Pi)

```sh
cd ~/API-Rpi16GB/docker
docker compose up --build -d
```

Docker downloads Zig, compiles the backend, and starts all services. See [docs/setup.md](docs/setup.md) for the full first-time setup and background notes.

### First-time setup notes
Before first start on the Pi:
1. generate secrets
2. generate TLS certificates
3. start the stack
4. run the password setup script

Example:
```sh
cd ~/API-Rpi16GB
sudo sh docker/setup_tls.sh

cd docker
docker compose up --build -d
sh ./set_passwords.sh
```
The setup_tls.sh script generates CA and service certificates for nginx and Mosquitto. The set_password.sh script sets PostgreSQL user passwords and generates the Mosquitto password file.

## End-to-End Test

A successful ent-to-end - MQTT → API → database - test can be done with:

```sh
docker compose exec mosquitto mosquitto_pub \
  -h localhost \
  -p 8883 \
  --cafile /mosquitto/ssl/ca.crt \
  --insecure \
  -u sensor01 \
  -P sensorpass \
  -t sensor01/data \
  -m "{\"value\":22.4,\"unit\":\"C\"}"
```

Expected Controller log:

```
Logged in to Zig API via nginx
Connected to broker
Forwarded sensor01: 22.4 C via nginx to Zig API
```

Database check:

```sh
docker compose exec postgres psql -U postgres -d sensor \
-c "SELECT sensor_id, value, unit, recorded_at FROM sensor_data ORDER BY recorded_at DESC LIMIT 5;"
```

Expected result:

```
sensor01 | 22.4 | C
```

## Seeding Fake Data

The `seeder` service generates realistic sensor data (temperature + humidity)
to populate `sensor_data` and `sensor_data_archive` for dashboard demos and
schema testing. It is not started by `docker compose up` — it lives in the
`tools` profile and runs on demand.

The seeder uses a `demo-` sensor ID prefix so fake data is trivial to remove:

```sql
DELETE FROM sensor_data         WHERE sensor_id LIKE 'demo-%';
DELETE FROM sensor_data_archive WHERE sensor_id LIKE 'demo-%';
```

### How it routes rows

Rows are split based on their generated `recorded_at`:

- older than 7 days → inserted directly into `sensor_data_archive`
  with `archived_at = recorded_at + 7 days`
- within the last 7 days → inserted into `sensor_data`

Both tables are populated in a single transaction; no need to wait for the
archiver to fire.

### Realistic value generation

For each sensor, the seeder spreads rows evenly across the time window and
walks values forward chronologically. Each reading is the previous reading
plus a small mean-reverting random step, hard-capped at `max_delta_per_min`
per minute. A 60-second gap between samples can therefore never produce
more than that much change, so the data looks like a real room (slow drift
around a steady mean) instead of independent random samples.

Defaults assume a normally heated room: temp around 21.5 °C, humidity
around 45 %, max temp change capped at 0.25 °C per minute. Humidity uses
a wider cap internally because humidity moves faster than temperature.

### Anomalies

A configurable percentage of rows is forced to `mean + anomaly_magnitude`,
bypassing the per-minute cap on purpose so the spike is visibly anomalous.
The next non-anomaly reading walks back from the spike rather than
teleporting straight back to the mean.

Two flags control this:
- `--anomaly-percent` sets how much of the data is corrupted (0 disables).
- `--anomaly-magnitude` sets how far an anomaly is from the sensor's mean,
  in the sensor's own unit. Use a negative value for cold-side anomalies.

### Usage

From `docker/` on the Pi:

```sh
# defaults: 10 000 rows over 30 days, 0.5% anomalies at +15 from mean
docker compose --profile tools run --rm seeder

# stress-test the archive purge with > 3 years of data
docker compose --profile tools run --rm seeder --rows 200000 --days 1100

# multiple sensors, heavier corruption (5% of rows are anomalies)
docker compose --profile tools run --rm seeder \
    --sensors demo-temp-01,demo-temp-02,demo-humid-01 \
    --rows 50000 --anomaly-percent 5

# clean run with no anomalies and tighter per-minute realism
docker compose --profile tools run --rm seeder --anomaly-percent 0 --max-delta-per-min 0.1
```

### CLI options

| Flag | Default | Meaning |
|---|---|---|
| `--rows N`              | `10000` | total rows to insert |
| `--days N`              | `30`    | spread `recorded_at` across the last N days |
| `--sensors a,b,c`       | `demo-temp-01,demo-humid-01` | sensor IDs to generate for |
| `--anomaly-percent P`   | `0.5`   | percentage of rows that are anomalies; `0` disables |
| `--anomaly-magnitude M` | `15.0`  | how far an anomaly deviates from the mean, in the sensor's unit |
| `--max-delta-per-min F` | `0.25`  | cap on per-minute change for temp sensors (humidity scaled internally) |

Sensor IDs are classified by name: `*temp*` produces °C readings around
21.5 °C, `*humid*` produces % readings around 45 %, anything else produces
unitless values around 50.
